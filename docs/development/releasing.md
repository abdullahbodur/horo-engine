# Releasing Horo Engine

This repository ships releases manually from `main` through the GitHub Actions `Release` workflow.

## What the workflow does

- exposes a `workflow_dispatch` selection box for `patch`, `minor`, or `major`
- allows release publication only when the actor matches `github.repository_owner`
- requires the workflow to run from the current `origin/main` HEAD
- re-runs Linux, Windows, and macOS debug build-and-test validation before publishing
- creates an annotated `vX.Y.Z` tag
- publishes a GitHub Release with autogenerated notes

If no earlier semantic version tag exists yet, the first run bootstraps from the version already declared in `CMakeLists.txt`. The selected bump matters from the second release onward.

## Recommended repository setup

For defense in depth, protect the GitHub Actions `release` environment in the repository settings and add yourself as the required reviewer. The workflow already blocks non-owner actors, but the protected environment gives the release button a second explicit approval gate.

## How to cut a release

1. Merge the changes you want into `main`.
2. Open `Actions -> Release`.
3. Run the workflow on `main`.
4. Choose `patch`, `minor`, or `major`.
5. Wait for the cross-platform verification jobs to finish.
6. Approve the `release` environment if GitHub asks for it.
7. The workflow will push the tag and publish the GitHub Release.

## Current scope

The first version of the workflow publishes source releases and semantic version tags. It is intentionally structured so we can add binary packaging later by inserting platform packaging jobs before `publish-release` and uploading their artifacts into the same GitHub Release.
